# 4-24 | Bug Fix Prompt — Структурированный дебаг

> **Тип:** Instruction
> **Когда использовать:** Когда в проекте обнаружен баг и нужно его системно исправить
> **Результат:** Баг найден, исправлен, протестирован, регрессий нет

---

## Философия этого документа

Этот файл — **готовый промпт для Claude Code**. Вы не пишете его сами — копируете и используете.

**Принцип работы:**
- Вы — архитектор и коммуникатор
- Claude Code — исполнитель: читает промпт, делает работу
- Ваша задача — описать баг и скопировать правильный промпт

**Workflow:**
```
Вы → копируете промпт → вставляете в Claude Code (VS Code)
       ↓
       прикрепляете контекст бага (шаги, логи, ожидание vs реальность)
       ↓
Claude Code → идёт по 6 шагам: reproduce → root cause → fix → regress-тест
       ↓
Вы → принимаете исправление → убеждаетесь, что регрессий нет
```

Промпт уже отлажен. Не нужно его улучшать с первого раза — просто используйте.

---

## Что это такое

Структурированный процесс дебага из 6 шагов. Предотвращает антипаттерн "слепого ретрая" — когда Claude Code (или человек) просто пробует разные исправления без понимания причины.

**Антипаттерн "слепой ретрай":**
```
"Не работает" → попробуй это → не помогло → попробуй то → не помогло →
→ переписал всё с нуля → сломал другое → потратил 2 часа
```

**Наш подход:**
```
Воспроизведи → Диагностируй → Гипотеза → Целевое исправление → Тест → Проверка регрессий
```

## Промпт для копирования

```
У меня баг: [опиши симптом — что происходит вместо ожидаемого]

Исправь его по этому алгоритму. НЕ ПРОПУСКАЙ ШАГИ.

## Шаг 1: Воспроизведение
- Определи точные шаги для воспроизведения бага
- Зафиксируй: что происходит (actual) vs что ожидается (expected)
- Проверь: баг стабильный или плавающий?
- Покажи ошибку из консоли/логов если есть

## Шаг 2: Диагностика
- Найди файл(ы) где живёт проблемный код
- Проследи поток данных: откуда приходят данные → где обрабатываются → где отображаются
- Определи: это ошибка фронтенда, бэкенда, базы данных, или интеграции?
- Используй git log чтобы понять когда появился баг (git log --oneline -20)

## Шаг 3: Гипотеза
- Сформулируй ОДНУ конкретную гипотезу: "Баг возникает потому что..."
- Объясни ПОЧЕМУ ты так думаешь (покажи код, который подтверждает гипотезу)
- Если гипотеза не подтверждается — сформулируй следующую, не угадывай

## Шаг 4: Целевое исправление
- Исправь ТОЛЬКО то, что вызывает баг
- Не рефактори "заодно" — это создаёт риск регрессий
- Минимальное изменение = меньше шансов сломать другое
- Покажи diff: что было → что стало

## Шаг 5: Тестирование
- Повтори шаги воспроизведения из Шага 1
- Баг больше не воспроизводится?
- Проверь граничные случаи: пустые данные, большие данные, спецсимволы

## Шаг 6: Проверка регрессий
- Проверь связанные функции (те же файлы, те же данные)
- Запусти tsc --noEmit — нет ли новых TypeScript ошибок
- Проверь что другие фичи не сломались

## Формат ответа
Покажи результат каждого шага, включая:
- Шаг 1: скриншот/текст ошибки и шаги воспроизведения
- Шаг 2: путь к проблемному файлу + строка
- Шаг 3: "Гипотеза: [текст]"
- Шаг 4: diff исправления
- Шаг 5: "Тест: [PASS/FAIL]"
- Шаг 6: "Регрессии: [нет / найдены]"

Коммит: "fix: [краткое описание бага]"
```

## Пример: TaskFlow — задача не сохраняется

### Симптом
Пользователь заполняет форму новой задачи, нажимает "Создать" — форма очищается, но задача не появляется в списке. Ошибок в UI нет.

### Пошаговое решение

**Шаг 1: Воспроизведение**
```
1. Открыть /tasks
2. Нажать "Новая задача"
3. Заполнить: название "Тестовая задача", проект "TaskFlow", приоритет "Высокий"
4. Нажать "Создать"
5. Actual: форма очищается, задача НЕ появляется в списке
6. Expected: задача появляется в списке
7. Баг стабильный — воспроизводится каждый раз
8. Консоль браузера: нет ошибок
9. Network tab: POST запрос возвращает 200, но body пустой
```

**Шаг 2: Диагностика**
```
Поток данных:
1. Форма → src/components/tasks/TaskForm.tsx (handleSubmit)
2. Server Action → src/app/actions/tasks.ts (createTask)
3. Supabase → таблица tasks (INSERT)
4. Revalidate → src/app/(dashboard)/tasks/page.tsx (перезагрузка списка)

Проверяю Server Action:
```
```typescript
// src/app/actions/tasks.ts — строка 15
export async function createTask(formData: FormData) {
  const supabase = await createClient()

  const { data, error } = await supabase
    .from('tasks')
    .insert({
      title: formData.get('title'),
      project_id: formData.get('project_id'),
      priority: formData.get('priority'),
      user_id: formData.get('user_id'),  // ← ПОДОЗРИТЕЛЬНО
    })
    .select()
    .single()

  if (error) throw error
  revalidatePath('/tasks')
  return data
}
```
```
Проблема найдена: user_id берётся из formData (клиент), а не из auth.
Supabase RLS-политика требует user_id = auth.uid().
Клиент не отправляет user_id → INSERT проваливается из-за RLS → ошибка проглатывается.
```

**Шаг 3: Гипотеза**
```
Гипотеза: INSERT не проходит потому что user_id = null.
RLS-политика на tasks требует user_id = auth.uid(), но user_id не передаётся из формы.
Supabase возвращает ошибку, но throw error не ловится на клиенте
потому что Server Action бросает исключение, а форма не обрабатывает rejected promise.
```

**Шаг 4: Целевое исправление**
```typescript
// src/app/actions/tasks.ts — ИСПРАВЛЕНИЕ

export async function createTask(formData: FormData) {
  try {
    const supabase = await createClient()
    const { data: { user } } = await supabase.auth.getUser()

    if (!user) {
      return { success: false, error: 'Необходимо авторизоваться' }
    }

    const { data, error } = await supabase
      .from('tasks')
      .insert({
        title: formData.get('title') as string,
        project_id: formData.get('project_id') as string,
        priority: formData.get('priority') as string,
        user_id: user.id,  // ← ИСПРАВЛЕНО: берём из auth
      })
      .select()
      .single()

    if (error) {
      return { success: false, error: error.message }
    }

    revalidatePath('/tasks')
    return { success: true, data }
  } catch (e) {
    return { success: false, error: 'Не удалось создать задачу' }
  }
}
```

```typescript
// src/components/tasks/TaskForm.tsx — обработка ответа

const handleSubmit = async (formData: FormData) => {
  const result = await createTask(formData)
  if (result.success) {
    setOpen(false)  // Закрыть модал
  } else {
    setError(result.error)  // Показать ошибку пользователю
  }
}
```

**Шаг 5: Тестирование**
```
1. Повторил шаги воспроизведения — задача создалась и появилась в списке ✓
2. Проверил с пустым названием — показывает ошибку валидации ✓
3. Проверил с длинным названием (500 символов) — работает ✓
4. Проверил без авторизации — показывает "Необходимо авторизоваться" ✓

Тест: PASS
```

**Шаг 6: Проверка регрессий**
```
- Редактирование задачи — работает ✓
- Удаление задачи — работает ✓
- Фильтрация по проекту — работает ✓
```

## Частые категории багов (Next.js + Supabase)

| Категория | Типичный симптом | Где искать |
|-----------|-----------------|------------|
| RLS блокирует запрос | Данные не возвращаются, ошибок нет | Supabase → Policies |
| user_id не передаётся | INSERT/UPDATE молча проваливается | Server Actions → auth.getUser() |
| Кэш Next.js | Старые данные после мутации | revalidatePath / revalidateTag |
| Типы не совпадают | Runtime ошибка при доступе к полю | TypeScript interfaces vs БД |
| Env переменные | undefined на сервере или клиенте | .env.local, NEXT_PUBLIC_ prefix |
| Tailwind v4 классы | Стили не применяются | Проверь v3→v4 breaking changes |
| Middleware redirect loop | Страница не загружается, бесконечный редирект | src/middleware.ts matcher |
| Supabase client на клиенте | Auth не работает | createBrowserClient vs createServerClient |

## Когда НЕ использовать этот промпт

- **Баг воспроизводится только локально** и связан с окружением → проверь версии Node.js, npm, переменные окружения
- **Баг в сторонней библиотеке** → проверь GitHub Issues, обнови версию или найди workaround
- **"Баг" — это непонятное поведение** → возможно, это работает как задумано, перечитай спецификацию
