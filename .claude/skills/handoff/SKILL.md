# Скилл: Handoff между сессиями

## Триггер
Пользователь говорит "сделай handoff" / "запиши хендовер" / "до встречи" / `/handoff`.

## Что делать

Создать/обновить `HANDOVER.md` в корне проекта со всем критичным контекстом для следующей сессии. Это файл, который ты прочитаешь первым в начале новой сессии — он экономит время на восстановление контекста.

## Шаблон HANDOVER.md

```markdown
# HANDOVER — КРЕСТ
> Дата: YYYY-MM-DD HH:MM | Сессия: краткое описание

---

## Текущий статус

✅ Что работает:
- [перечислить рабочие фичи]

🔄 Что в процессе:
- [то что начали но не закончили]

❌ Что сломано:
- [известные баги с location:line]

---

## Главные решения этой сессии

| Решение | Почему | Файлы |
|---------|--------|-------|
| [что сделали] | [обоснование] | [где смотреть] |

---

## Что прочитать первым в новой сессии

1. `HANDOVER.md` (этот файл)
2. `CLAUDE.md` (всегда)
3. `SPEC.md` если работаем над спецификацией
4. memory/MEMORY.md — индекс памяти

## Команды для быстрого старта

\`\`\`bash
git status
git log --oneline -5
\`\`\`
```

## Что ВКЛЮЧАТЬ в handoff

- ✅ Текущее состояние (рабочее/сломанное)
- ✅ Решения с обоснованием
- ✅ Активные TODO с конкретикой (file:line)
- ✅ Важные креды (где их искать, не сами креды)
- ✅ Гипотезы и контекст принятых решений

## Что НЕ ВКЛЮЧАТЬ

- ❌ Переписку всей сессии (это есть в jsonl логе)
- ❌ Сами секреты/токены/пароли
- ❌ Полный код изменений (это есть в git diff)
- ❌ Информацию которая в CLAUDE.md / SPEC.md / памяти

## После создания HANDOVER

1. `git add HANDOVER.md`
2. Опционально: `git commit -m "docs: handoff [дата]"`
3. Сообщить пользователю: "Хендовер записан. До встречи!"
