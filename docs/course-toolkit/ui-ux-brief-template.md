# 2-16 | UI/UX Brief — Справочник структуры

> Reference-документ: что Claude генерирует для UI/UX Brief.
> Используйте для валидации результата и формулирования требований — не для ручного заполнения.

---

## Философия этого документа

**Вы не пишете UI/UX Brief вручную.** Вы показываете Claude AI референсы (скриншоты Linear/Notion/Vercel Dashboard), описываете характер интерфейса и аудиторию — а Claude генерирует полный бриф с цветовой схемой, типографикой, экранами, компонентами и анимациями.

Зачем тогда этот справочник?

1. **Валидация** — понять, все ли разделы Claude заполнил (цвета, типографика, экраны, breakpoints, состояния)
2. **Корректировка** — если палитра не соответствует референсам или нет shadcn/ui компонентов — знать как попросить доработать
3. **Коммуникация** — говорить с Claude на языке "сделай тёмно-фиолетовый primary" вместо "перекрась"
4. **Контроль качества** — убедиться, что UI_UX_BRIEF.md пригоден для Claude Code

**Правильный workflow:**
```
Вы → референсы (скриншоты) + описание характера ("минималистичный dark")
Claude AI → генерирует UI_UX_BRIEF.md по этому формату
Вы → используете этот справочник чтобы валидировать результат
Claude → дорабатывает палитру/компоненты/анимации по вашим замечаниям
Claude Code → использует бриф как правило при создании каждого компонента
```

**Правило:** НИКОГДА не позволяйте Claude придумывать дизайн без референсов. Всегда покажите 2-3 приложения с похожим стилем — это ключ к хорошему результату.

---

## Что это и зачем

UI/UX Brief — документ, который превращает абстрактное "сделай красиво" в конкретные технические требования к интерфейсу. Без него frontend-developer (или Claude Code) будет принимать решения наугад, и каждый экран будет выглядеть по-разному.

Этот документ генерируется ОДИН раз для проекта и становится единым источником истины для всех UI-решений: цвета, шрифты, отступы, компоненты, анимации, адаптивные точки.

AI-Архитектор валидирует бриф и загружает его в проект Claude AI вместе со спецификацией. Claude Code использует этот бриф как правило при создании каждого компонента.

## 7. Адаптивные точки (breakpoints)

Tailwind v4 дефолтные breakpoints:
- sm: 640px — мобильный ландшафт
- md: 768px — планшет портрет
- lg: 1024px — планшет ландшафт / маленький ноутбук
- xl: 1280px — десктоп
- 2xl: 1536px — большой десктоп

### Поведение на breakpoints:
- < md: [скрыть sidebar, один столбец, мобильная навигация]
- md-lg: [компактный sidebar или drawer, два столбца]
- > lg: [полный sidebar, многоколоночный лейаут]

## 8. Анимации и переходы

### Общие принципы
- Длительность: 150-300ms (быстрые, не замедляют работу)
- Easing: ease-out для появления, ease-in для исчезновения
- Уважай `prefers-reduced-motion` — отключай анимации

### Конкретные анимации
| Элемент | Анимация | CSS/Tailwind |
|---------|----------|-------------|
| Модалки | Fade in + scale from 95% | transition-all duration-200 |
| Sidebar mobile | Slide from left | transition-transform duration-300 |
| Skeleton загрузка | Pulse | animate-pulse |
| Toast уведомления | Slide from right | animate-in slide-in-from-right |
| Hover на кнопках | Фон + scale | hover:bg-primary-hover transition-colors |
| Drag-and-drop | Поднятие + тень | shadow-lg scale-105 |

---

## 9. Состояния UI

### Каждый экран должен обрабатывать:

1. **Loading** — скелетон или спиннер (не пустой экран)
2. **Empty** — иллюстрация + CTA ("Создайте первый проект")
3. **Error** — сообщение + кнопка повтора
4. **Success** — toast-уведомление (не отдельный экран)

### Формы:
- Валидация: inline под каждым полем (красный текст)
- Кнопка отправки: disabled + спиннер при loading
- Успех: toast + редирект (или закрытие модалки)
- Ошибка: toast + подсветка проблемного поля

## Чек-лист валидации (после генерации)

Когда Claude сгенерировал UI_UX_BRIEF.md — пройдитесь по этому чек-листу. Если что-то не так, попросите Claude доработать конкретный раздел.

```
[ ] 1. Концепция описана с отсылкой к референсам (не "современный", а "как Linear")

[ ] 4. Типографика: указан шрифт (Inter/Geist) и размеры с Tailwind-классами

[ ] 6. Все экраны перечислены с путями и компонентами
[ ] 7. Для каждого экрана — состояния: loading, empty, error, data
[ ] 8. Адаптивное поведение описано для mobile (<md), tablet (md-lg), desktop (>lg)
```

## Workflow: как создать UI_UX_BRIEF.md за 30 минут

```
ШАГ 1: Соберите референсы
  ↓ 2-3 скриншота приложений с похожим стилем
  ↓ Краткое описание характера ("минималистичный dark-first dashboard")
  
ШАГ 2: Откройте Claude AI (web)
  ↓ Загрузите 4 артефакта:
  • Скриншоты референсов (Linear, Notion, Vercel Dashboard и т.д.)
  • Ваше описание характера интерфейса и аудитории
  • SPEC.md проекта (чтобы Claude знал список экранов)
  • 2-16-UI_UX_BRIEF_TEMPLATE.md (этот справочник)
  
ШАГ 3: Промпт:
  "Сгенерируй UI_UX_BRIEF.md по формату из справочника. 
   Цвета и стиль — из моих референсов. Экраны — из SPEC.md.
   Используй Tailwind v4 (@theme, shadow-xs, outline-hidden).
   Заполни все 10 разделов: цвета, типографика, лейаут, экраны, 
   компоненты, breakpoints, анимации, состояния."
  
ШАГ 4: Получите результат
  ↓ Claude вернёт UI_UX_BRIEF.md на 200-400 строк
  
ШАГ 5: Валидация (этот документ)
  ↓ Пройдитесь по чек-листу выше
  
ШАГ 6: Если нужны правки — попросите Claude
  "В разделе [X] [Y] не соответствует моему референсу. Перепиши."
  
ШАГ 7: Сохраните UI_UX_BRIEF.md в корне проекта
  ↓ Рядом с SPEC.md и PROJECT_IDEA.md
  
ШАГ 8: Передайте в 3-17 Claude Code Setup Generator
  ↓ Бриф станет частью rules для frontend-developer агента
```

**Итог:** вы не написали UI/UX Brief руками. Вы показали референсы, валидировали сгенерированный документ, дали правки. Это и есть роль архитектора.
