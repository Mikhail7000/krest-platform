## 3. Домашнее задание на неделю (9 пунктов)

Каждый пункт — отдельный **assignment** с собственным типом submission и workflow одобрения куратором.

### Пункт 1. Посмотреть видео «Малый крест»

- **Тип:** просмотр (auto-complete после первого full watch)
- **Модерация:** не требуется
- **Заметка:** «просмотр вместе с куратором» — пока **отложено** (нет ясности как реализовать), ученик смотрит сам

### Пункт 2. Составить подробный конспект

- **Тип:** текстовое поле + кнопка «Отправить куратору»
- **Submission:** длинный текст, без ограничения
- **Workflow куратора:** одобрить / отказать / вернуть на доработку / комментарий / оценка

### Пункт 3. Писать «Малый крест» каждый день (переписывать)

- **Тип:** ежедневная сдача — **фото** переписанного текста
- **Submission:** загрузка фото (можно несколько за день — листы)
- **Период:** все 7 дней недели → отдельный submission на каждый день
- **Хранилище:** Supabase Storage, bucket `student-submissions/{user_id}/block-{n}/photos/`
- **Workflow куратора:** одобрить день / коммент

### Пункт 4. Выучить местописания

- **Тип:** ежедневная сдача — **видеокружок** (запись с камеры, single-take, нельзя загрузить готовый файл)
- **Submission:** видео с камеры (MediaRecorder API), запись на месте
- **Fallback:** если запись с камеры не поддерживается на устройстве — обычная загрузка видео
- **Период:** ежедневно
- **Хранилище:** Supabase Storage, bucket `student-submissions/{user_id}/block-{n}/scripture-videos/`
- **Workflow куратора:** одобрить / коммент

**Технический нюанс:** Telegram «video message» (кружок) изнутри WebApp нативно записать **нельзя** — это API клиента Telegram. Решение: MediaRecorder с фронта пишет квадратное видео, отображается как кружок, отправляется как файл. Запись «с первого дубля» обеспечивается тем, что **input file отключаем**, доступна только кнопка «Записать».

### Пункт 5. Молитва по кресту каждый день

- **Тип:** ежедневный чекин «молился сегодня — да/нет» + опц. короткая заметка
- **Submission:** boolean + текст
- **Период:** ежедневно
- **Workflow куратора:** просмотр (без одобрения), куратор видит дисциплину

### Пункт 6. Сдать малый крест куратору (пересказ)

- **Тип:** запись **видео-пересказа** (как в видеоуроке, максимально точно)
- **Submission:** видео-запись через MediaRecorder
- **Workflow куратора:** одобрить / отказать / вернуть с комментарием / оценка
- **Это ключевой gate — без одобрения нельзя перейти к следующему блоку** (см. п.7 ниже)

### Пункт 7. Практика «Эпоха пятницы»

- **Тип:** комплексный отчёт о практике на улице
- **Submission:** несколько полей —
  - Конспект практики (текст)
  - Фото/видео-вложения с места
  - Свободная форма «как прошла эпоха пятницы»
- **Workflow куратора:** одобрить / коммент

### Пункт 8. Эмоции и свидетельства

- **Тип:** отдельный раздел-журнал, текст + опц. медиа
- **Submission:** заметки, можно несколько
- **Workflow куратора:** просмотр (без обязательного одобрения)

### Пункт 9. Ежедневный отчёт

- **Тип:** структурированный ежедневный шаблон-отчёт (отдельная ветка)
- **Submission:** заполняемая форма (поля TBD по примеру существующих чатов)
- **Период:** ежедневно
- **Workflow куратора:** просмотр

## 4. Gate: переход к следующему блоку

Блок считается «сдан» когда:
- ✅ Видео просмотрено (Пункт 1)
- ✅ Конспект одобрен куратором (Пункт 2)
- ✅ Все местописания заучены (Пункт 4) — куратор подтвердил
- ✅ Пересказ одобрен куратором (Пункт 6) — главный gate
- ✅ Отчёт по «Эпохе пятницы» одобрен (Пункт 7)

После одобрения всех gate-пунктов:
- `student_progress.admin_approved = true`
- `profiles.blocks_unlocked = LEAST(blocks_unlocked + 1, 10)`
- Push в Telegram: «Поздравляю! Тебе открыт блок N: НАЗВАНИЕ» + красиво отформатированный конспект следующего блока

Пункты 3, 5, 8, 9 (ежедневные) — **продолжают вестись** даже после открытия следующего блока (это привычки, не gate-задания).

## 5. Роли и привязки

### 5.1 Админы

- **1 главный админ** (Михаил, chat_id 255214568) — root
- Главный админ может **назначать других админов** через UI: добавить chat_id или email → роль `admin`
- Любой админ может **назначать кураторов** (роль `curator` или `leader`)
- Передача root-управления — отдельная функция (не сейчас)

### 5.2 Кураторы (= лидеры)

- Каждый ученик **привязан ровно к одному куратору** (`profiles.curator_id`)
- Куратор видит только своих учеников
- Куратор получает push в Telegram при каждом новом submission от своих учеников
- Главный админ — суперпользователь, видит всех

### 5.3 Привязка ученика к куратору

- Привязка происходит при регистрации (TBD: либо ручное назначение админом, либо автоматическое распределение)
- Можно перепривязать через админ-панель

## 6. Workflow куратора по каждому submission

Стандартный набор действий для всех типов submission:

| Действие | Эффект |
|----------|--------|
| ✅ Одобрить | submission закрыт, прогресс зачтён |
| ❌ Отказать | submission закрыт без зачёта, ученик получает push с причиной |
| ↩️ Вернуть на доработку | submission остаётся открытым, ученик может пересдать |
| 💬 Комментарий | свободный текст, ученик видит |
| ⭐ Оценка | 1–5 звёзд (опц.) |

Все действия → push в Telegram ученику.
