# 3-20 | CLAUDE.md — Справочник структуры

> Reference-документ: как устроен CLAUDE.md, который Claude генерирует для вашего проекта.
> Используйте для валидации результата и формулирования требований — не для ручного заполнения.

---

## Философия этого документа

**Вы не пишете CLAUDE.md вручную.** Claude Code сам создаёт его из вашего описания проекта (PROJECT_IDEA.md или SPEC.md).

Зачем тогда этот справочник?

1. **Валидация** — понять, правильно ли Claude построил структуру
2. **Корректировка** — если что-то не так, знать как попросить изменить
3. **Коммуникация** — говорить с Claude на языке "перенеси это из CLAUDE.md в rules" вместо "это не то"
4. **Контроль качества** — проверить, что CLAUDE.md не превышает лимиты и содержит всё необходимое

**Правильный workflow:**
```
Вы → описываете проект (PROJECT_IDEA.md)
Claude → генерирует CLAUDE.md + субагентов + rules (через Setup Generator)
Вы → используете этот справочник чтобы валидировать результат
Claude → вносит правки по вашим замечаниям
```

## Что такое CLAUDE.md

CLAUDE.md — это инструкция для Claude Code, которая лежит в корне проекта. Claude автоматически читает этот файл при старте сессии и следует указаниям в нём.

**Аналогия:** CLAUDE.md — это бриф для подрядчика. Чем точнее бриф, тем меньше переделок.

### Ограничение по размеру

**Максимум 120 строк.** Это не произвольное ограничение — длинный CLAUDE.md:
- Съедает контекстное окно (токены, за которые вы платите)
- Размывает фокус Claude — он пытается следовать всему и не следует ничему
- Дублирует информацию из субагентов и rules

Правило: в CLAUDE.md — только то, что нужно КАЖДОМУ агенту ВСЕГДА.
Детали — в `.claude/agents/`, `.claude/rules/`, `.claude/skills/`.

---

## Структура CLAUDE.md: 8 секций

```
1. Обзор проекта          (5-8 строк)
2. Стек технологий         (5-8 строк)
3. Архитектура             (15-25 строк)
4. Правила для БД          (5-8 строк)
5. Правила кодирования     (5-8 строк)
6. MCP-серверы              (3-5 строк)
7. Субагенты               (5-10 строк)
8. Команды                 (5-8 строк)
```

Общий итог: 48-80 строк основного контента + заголовки и пустые строки = до 120 строк.

## Секция 1: Обзор проекта

**Что Claude генерирует:** Одно предложение — что это, для кого, ключевая ценность. Плюс 3-5 ключевых функций.

**Откуда берёт:** из раздела "Проблема" и "Аудитория" вашего PROJECT_IDEA.md.

**Как проверить:** "трекер задач для фрилансеров" vs "CRM для B2B-продаж" — это разные решения. Если обзор размыт — попросите Claude переделать конкретнее.

```markdown
## Обзор
[Название] — [что это] для [кого]. [Ключевая ценность в одном предложении].

Ключевые функции:
- [Функция 1]
- [Функция 2]
- [Функция 3]
```

---

## Секция 2: Стек технологий

**Что Claude генерирует:** Только реально используемые технологии. Без "может пригодиться".

**Как проверить:** совпадает ли стек с тем, что в SPEC.md? Нет ли устаревших версий (Next.js 15 вместо 16, Tailwind v3 вместо v4)? Если Claude подтянул старые версии — попросите обновить через Context7.

```markdown
## Стек технологий
- Frontend: [фреймворк], [язык], [UI-библиотека]
- Backend: [БД], [auth], [API-подход]
- Платежи: [платёжная система]
- Деплой: [хостинг фронта], [хостинг бэка]
```

## Секция 3: Архитектура

**Что Claude генерирует:** Дерево файлов проекта — только ключевые директории и файлы. Это карта навигации для всех агентов.

**Правило:** структура до 2-го уровня вложенности. Детали папок — в `.claude/agents/`.

**Как проверить:** отражает ли дерево реальную структуру после scaffold'а? Если Claude создал проект, но CLAUDE.md показывает другую структуру — попросите синхронизировать с реальной.

## Секция 4: Правила для БД

**Что Claude генерирует:** Конвенции именования, обязательные практики. Коротко — детали в `agents/database-architect.md`.

**Как проверить:** есть ли упоминание RLS (обязательно), есть ли путь к миграциям, указан ли способ генерации типов? Если чего-то не хватает — попросите добавить.

```markdown
## Работа с Supabase
- Изменения схемы — ТОЛЬКО через миграции в supabase/migrations/
- RLS обязательна на каждой таблице
- Типы: npx supabase gen types typescript > src/types/database.ts
- snake_case для таблиц и колонок
```

---

## Секция 5: Правила кодирования

**Что Claude генерирует:** Универсальные правила, применимые ко всем файлам проекта.

**Как проверить:** правила должны быть конкретными ("запрещён any", "максимум 200 строк"), а не абстрактными ("пиши чистый код"). Если Claude генерит абстракции — попросите конкретизировать с числами и примерами.

```markdown
## Правила кодирования
- TypeScript strict mode, запрещён any
- camelCase переменные, PascalCase компоненты
- Один компонент = один файл, максимум 200 строк
- Импорты через @/ алиас
- Server Components по умолчанию, 'use client' только для интерактива
- Обработка ошибок: try/catch для async, Error Boundaries для UI
```

---

## Секция 6: MCP-серверы

**Что Claude генерирует:** Какие MCP подключены и когда их использовать. БЕЗ токенов и ключей.

**Как проверить:** нет ли в файле секретов (ключи/токены в CLAUDE.md — безопасная угроза). Если есть — немедленно уберите и скажите Claude: "переместить секреты в .env". Context7 должен быть обязательным для всех проектов с библиотеками.

```markdown
## MCP
- Context7: ВСЕГДА использовать перед написанием кода с Next.js, Supabase, Tailwind
- Supabase: подключён (ref: abc123example)
- GitHub: подключён
```

---

## Секция 7: Субагенты

**Что Claude генерирует:** Список субагентов с кратким описанием. Детали — в `.claude/agents/`.

**Как проверить:** соответствует ли состав команды типу проекта (см. справочник "Архитектура субагентов")? Назначены ли модели (Opus для архитектуры, Sonnet для реализации)? Если состав не подходит — попросите: "пересмотри команду под наш тип проекта".

```markdown
## Субагенты
- database-architect (Opus) — схема БД, миграции, RLS, индексы
- backend-engineer (Sonnet) — API, бизнес-логика, интеграции
- frontend-developer (Sonnet) — UI-компоненты, страницы, стилизация
- qa-reviewer (Sonnet) — тестирование, ревью, безопасность
```

---

## Секция 8: Команды

**Что Claude генерирует:** Основные команды для разработки, сборки и деплоя.

**Как проверить:** совпадают ли команды с тем, что в `package.json`? Если Claude придумал команды, которых нет — попросите синхронизировать.

```markdown
## Команды
- npm run dev — разработка (Turbopack)
- npm run build — сборка
- npm run lint — линтер
- npx supabase db push — применить миграции
```

## Полный пример: что Claude сгенерирует для TaskFlow

Ниже — пример того, что Claude создаст после того, как вы загрузите PROJECT_IDEA.md TaskFlow в Setup Generator. Используйте как образец для сравнения с тем, что получите в своём проекте.

```markdown
# TaskFlow — Трекер задач

## Обзор
TaskFlow — веб-приложение для управления задачами фрилансеров. Позволяет вести проекты, отслеживать дедлайны, выставлять счета клиентам и управлять подпиской.

Ключевые функции:
- Канбан-доска задач с дедлайнами и приоритетами
- Учёт времени по задачам (тайм-трекер)
- Выставление счетов клиентам (интеграция ЮKassa)
- Дашборд статистики: загрузка, доход, просрочки
- Подписочная модель: бесплатно (3 проекта) / про (безлимит)

## Стек технологий
- Frontend: Next.js 16 (App Router, Turbopack), TypeScript, Tailwind CSS v4, shadcn/ui
- Backend: Supabase (PostgreSQL, Auth, RLS, Realtime)
- Платежи: ЮKassa (подписки, разовые платежи)
- Деплой: Vercel (фронт), Supabase Cloud (БД)

## Архитектура
\```
src/
├── app/
│   ├── page.tsx                   # Landing page
│   ├── (auth)/                    # Login, Register
│   ├── (dashboard)/
│   │   ├── page.tsx               # Дашборд статистики
│   │   ├── projects/              # Список проектов
│   │   ├── projects/[id]/         # Канбан задач проекта
│   │   ├── time/                  # Тайм-трекер
│   │   ├── invoices/              # Счета клиентам
│   │   └── settings/              # Профиль, подписка
│   └── api/
│       ├── tasks/                 # CRUD задач
│       ├── projects/              # CRUD проектов
│       ├── invoices/              # Генерация счетов
│       └── payments/
│           └── webhook/           # Webhook ЮKassa
├── components/
│   ├── ui/                        # shadcn/ui компоненты
│   └── features/
│       ├── KanbanBoard.tsx        # Канбан-доска
│       ├── TimeTracker.tsx        # Тайм-трекер
│       ├── InvoiceForm.tsx        # Форма счёта
│       └── StatsCards.tsx         # Карточки статистики
├── lib/
│   ├── supabase/                  # Клиенты (server, browser)
│   ├── yukassa/                   # Интеграция ЮKassa
│   └── utils/                     # Форматирование, хелперы
└── types/
    ├── database.ts                # Авто-генерация из Supabase
    └── index.ts                   # Общие типы
\```

## Работа с Supabase
- Изменения схемы — ТОЛЬКО через миграции в supabase/migrations/
- RLS обязательна на каждой таблице
- Типы: npx supabase gen types typescript > src/types/database.ts
- snake_case для таблиц и колонок

## Правила кодирования
- TypeScript strict mode, запрещён any
- camelCase переменные, PascalCase компоненты
- Один компонент = один файл, максимум 200 строк
- Импорты через @/ алиас
- Server Components по умолчанию, 'use client' только для интерактива
- Обработка ошибок: try/catch для async, Error Boundaries для UI

## MCP
- Context7: ВСЕГДА использовать перед написанием кода с Next.js, Supabase, Tailwind
- Supabase: подключён (ref: abc123example)
- GitHub: подключён

## Субагенты
- database-architect (Opus) — схема БД, миграции, RLS, индексы
- backend-engineer (Sonnet) — API routes, server actions, ЮKassa
- frontend-developer (Sonnet) — канбан, тайм-трекер, дашборд
- qa-reviewer (Sonnet) — RLS-тесты, API-тесты, TypeScript strict

## Команды
- npm run dev — разработка (Turbopack)
- npm run build — сборка
- npm run lint — линтер
- npx supabase db push — применить миграции
- npx supabase gen types typescript > src/types/database.ts — обновить типы
```

## Антипаттерны: что НЕ должно быть в CLAUDE.md

Если Claude сгенерировал что-то из этой таблицы — попросите перенести/убрать.

| Чего быть не должно | Почему | Куда переместить |
|---------------------|--------|------------------|
| Подробные SQL-запросы | Засоряют контекст каждой сессии | `.claude/agents/database-architect.md` |
| Полные примеры компонентов | Каждый агент будет читать ненужное | `.claude/agents/frontend-developer.md` |
| API-документация | Слишком объёмно | `.claude/rules/api.md` |
| Токены и ключи | Небезопасно + бесполезно для Claude | `.env` + `.claude/settings.json` |
| TODO-листы | CLAUDE.md — конфиг, не таск-трекер | SPEC.md или GitHub Issues |
| История решений | "Раньше использовали X, но..." | Нигде — просто удалить |
| Абстрактные пожелания | "Пиши чистый код" — бессмысленно | Заменить на конкретные правила с числами |

**Промпт для исправления:**
```
В CLAUDE.md нашёл [проблема]. Перенеси это в [куда] и убери из CLAUDE.md.
Сохрани лимит 120 строк.
```

## Связь CLAUDE.md с остальными файлами

```
CLAUDE.md                      ← Общий контекст (120 строк max)
    │
    ├── .claude/agents/*.md    ← Детальные инструкции для каждого субагента
    │
    ├── .claude/rules/*.md     ← Правила с glob-паттернами (применяются к файлам)
    │
    ├── .claude/skills/*/      ← Повторяющиеся задачи (скиллы)
    │
    └── SPEC.md / SPEC_TEMPLATE.md  ← Спецификации фич
```

**Принцип:** CLAUDE.md — это оглавление книги. Агенты, правила и скиллы — это главы. Не нужно пересказывать содержание глав в оглавлении.

## Чек-лист валидации (после генерации)

Когда Claude сгенерировал CLAUDE.md — пройдитесь по этому чек-листу. Если что-то не так, попросите Claude исправить.

```
[ ] 1. CLAUDE.md лежит в корне проекта
[ ] 2. Размер не превышает 120 строк
[ ] 3. Все 8 секций присутствуют
[ ] 4. НЕТ токенов, ключей, паролей (это критично)
[ ] 5. НЕТ TODO и placeholder'ов
[ ] 6. Архитектура отражает реальную структуру папок
[ ] 7. Стек совпадает с package.json
[ ] 8. Субагенты перечислены с моделями (Opus/Sonnet)
[ ] 9. Context7 указан в секции MCP
[ ] 10. Версии библиотек актуальные (Next.js 16, Tailwind v4)
```

**Если найдена проблема — промпт для исправления:**
```
В CLAUDE.md проблема: [что не так].
Исправь и проверь, что лимит 120 строк сохраняется.
```

## Workflow: как создать CLAUDE.md за 5 минут

```
ШАГ 1: Напишите PROJECT_IDEA.md
  ↓ (используйте 2-10-PROJECT_IDEA_TEMPLATE как reference)
  
ШАГ 2: Откройте Claude AI (web)
  ↓ Загрузите 3 файла:
  • PROJECT_IDEA.md (ваш)
  • 3-17-CLAUDE_CODE_SETUP_GENERATOR.md (из toolkit)
  • 3-19-SUBAGENT_ARCHITECTURE_GUIDE.md (из toolkit, для контекста)
  
ШАГ 3: Промпт:
  "Сгенерируй полный пакет настройки Claude Code для этого проекта 
   по инструкциям из Setup Generator. Включи: CLAUDE.md, субагенты, rules, MCP."
  
ШАГ 4: Получите результат
  ↓ Claude вернёт CLAUDE.md + субагенты + rules + финальный промпт для Claude Code
  
ШАГ 5: Валидация (этот документ)
  ↓ Пройдитесь по чек-листу выше
  
ШАГ 6: Если нужны правки — попросите Claude
  "В CLAUDE.md секция [X] не отражает [Y]. Перепиши."
  
ШАГ 7: Скопируйте финальный промпт в Claude Code (VS Code)
  ↓ Claude Code создаст все файлы автоматически
```

**Итог:** вы не написали ни одной строки CLAUDE.md руками. Вы описали проект, валидировали результат, дали правки. Это и есть роль архитектора.
