# PROJECT_IDEA.md — Справочник структуры

> Reference-документ: что Claude генерирует для PROJECT_IDEA.md.
> Используйте для валидации результата и формулирования требований — не для ручного заполнения.

---

## Философия этого документа

**Вы не пишете PROJECT_IDEA.md вручную по пустому шаблону.** Вы описываете идею проекта Claude AI своими словами — а Claude структурирует её в PROJECT_IDEA.md по этому формату.

Зачем тогда этот справочник?

1. **Валидация** — понять, правильно ли Claude структурировал вашу идею
2. **Корректировка** — если раздел размыт или неточен, знать как попросить переписать
3. **Коммуникация** — говорить с Claude на языке "углуби раздел Цена проблемы" вместо "это не то"
4. **Контроль полноты** — проверить, что все 6 блоков заполнены конкретикой, а не абстракциями

**Правильный workflow:**
```
Вы → описываете идею (голос/текст, любой формат)
Claude → структурирует в PROJECT_IDEA.md по этому шаблону
Вы → используете этот справочник чтобы валидировать результат
Claude → вносит правки по вашим замечаниям
```

## Что проверять: конкретика vs абстракции

### Что Claude должен генерировать:
- Конкретные формулировки: "Пользователь загружает CSV-файл с задачами"
- Измеримые критерии: "Время создания задачи < 3 секунд"
- Реальные ограничения: "Бюджет 0 руб., только бесплатные инструменты"
- Конкретную аудиторию: "Фрилансеры-дизайнеры с 5+ проектами одновременно"

### Чего Claude НЕ должен генерировать (и как исправить):
- Абстракции: "Удобный интерфейс" (удобный — это КАК?) → попросите: "Замени на конкретные UX-метрики"
- Размытые цели: "Увеличить продуктивность" (на сколько? для кого?) → попросите: "Добавь числа"
- Технические решения: "использовать React" — это решает SPEC.md → попросите: "Убери тех-стек, он не сюда"
- Список фич на 20+ пунктов: если фич больше 7, это не MVP → попросите: "Сократи до 5 ключевых"

## Что Claude вернёт для TaskFlow

Ниже — пример того, что Claude сгенерирует, если вы опишете ему идею TaskFlow своими словами. Используйте как образец, с чем сравнивать свой результат.

```markdown
# TaskFlow — Трекер задач для фрилансеров

> Минималистичный трекер задач, который помогает фрилансерам управлять
> несколькими клиентскими проектами одновременно, не теряя дедлайны
> и не забывая выставить счёт.

---

## 1. Проблема

### Текущая ситуация (AS-IS)
Фрилансеры ведут задачи в Notion, Google Sheets, заметках в телефоне
или просто в голове. Для каждого клиента — отдельный файл или доска.
Время на проект трекают вручную или не трекают вообще.

### Боль
- Забывают про задачи и срывают дедлайны
- Не понимают, сколько времени реально потратили на проект
- Забывают выставить счёт за переработку
- Переключение между 5+ инструментами отнимает 30-40 минут в день

### Цена проблемы
Средний фрилансер-дизайнер теряет 1-2 клиентов в год из-за пропущенных
дедлайнов. При среднем чеке 50,000 руб. это 50-100K руб./год.
Невыставленные счета за переработку: ещё 20-30K руб./год.

---

## 2. Целевая аудитория

### Основная (ядро)
- **Кто:** Фрилансеры (дизайнеры, разработчики, копирайтеры) с 3+ клиентами одновременно
- **Размер:** ~500,000 активных фрилансеров в России (оценка по данным FL.ru и Хабр Фриланс)
- **Где обитают:** Telegram-чаты фрилансеров, Instagram, VC.ru, Хабр
- **Платёжеспособность:** Готовы платить 500-2,000 руб./мес за инструмент, который экономит время

### Вторичная (расширение)
- **Кто:** Маленькие агентства (2-5 человек), которые выросли из фриланса

### Антиаудитория (НЕ для них)
- Корпоративные команды 10+ человек — им нужен Jira/Asana
- Фрилансеры с 1 клиентом — им достаточно заметок

---

## 3. Решение

### Формула ценности
Фрилансер добавляет задачу за 10 секунд → видит все проекты и дедлайны
на одном экране → получает напоминание и автоматический расчёт времени.

### Ключевые функции (5 для MVP)

| # | Функция | Пользователь делает | Система делает | Результат |
|---|---------|---------------------|----------------|-----------|
| 1 | Быстрое добавление задачи | Пишет текст + выбирает проект | Парсит дедлайн из текста, присваивает приоритет | Задача создана за 10 сек |
| 2 | Канбан-доска по проектам | Перетаскивает карточки | Обновляет статус, логирует время перехода | Видит прогресс по каждому клиенту |
| 3 | Тайм-трекинг | Нажимает "Старт/Стоп" на задаче | Считает время, группирует по проекту | Знает, сколько часов потратил |
| 4 | Дедлайн-алерты | Ничего — система сама | Отправляет уведомление в Telegram за 24ч и 1ч до дедлайна | Не пропускает сроки |
| 5 | Отчёт для клиента | Нажимает "Сформировать отчёт" | Генерирует PDF: задачи, часы, итог | Выставляет счёт за реальное время |

### Что НЕ входит в MVP
- Интеграция с бухгалтерией и выставление счетов
- AI-приоритизация задач
- Мобильное приложение (только веб, адаптивный)
- Совместная работа команды

---

## 4. Ограничения

### Технические
- **Бюджет на инфраструктуру:** 0 руб. (free tier Supabase + Vercel)
- **Стек:** Next.js + Supabase (стандартный стек курса)
- **Интеграции:** Telegram Bot API (уведомления)

### Временные
- **Дедлайн MVP:** 2 недели
- **Доступность разработчика:** 2-3 часа в день

### Бизнесовые
- **Юрисдикция:** Россия
- **Платежи:** ЮKassa (подписка, подключим после валидации)
- **Персональные данные:** Да, email + имя → уведомление в Роскомнадзор

---

## 5. Критерии успеха

### MVP считается успешным, если:
1. 20 фрилансеров зарегистрировались за первые 2 недели после запуска
2. 5 из них создали 10+ задач (активные пользователи)
3. 3 из них вернулись через неделю (retention)

### Метрики для отслеживания:
- **Активация:** Создал хотя бы 1 проект и 3 задачи
- **Retention:** Зашёл на 7-й день после регистрации
- **Конверсия:** Из бесплатного в подписку (после запуска монетизации)

---

## 6. Монетизация

- **Модель:** Freemium (бесплатно до 3 проектов, подписка для безлимита)
- **Цена:** 990 руб./мес или 9,900 руб./год
- **Когда включать:** После 100 активных пользователей
```

## Чек-лист валидации (после генерации)

Когда Claude сгенерировал PROJECT_IDEA.md — пройдитесь по этому чек-листу. Если что-то не так, попросите Claude исправить.

```
[ ] 1. Все 6 разделов присутствуют (Проблема, Аудитория, Решение, Ограничения, Успех, Монетизация)
[ ] 2. Проблема описана с цифрами (цена проблемы в деньгах/времени)
[ ] 3. Целевая аудитория конкретна (не "все фрилансеры", а "дизайнеры с 3+ клиентами")
[ ] 4. Есть антиаудитория (кого продукт НЕ обслуживает)
[ ] 5. Формула ценности в одной фразе
[ ] 6. Ключевых функций — от 3 до 7 (не больше)
[ ] 7. Каждая функция разложена на "пользователь делает → система делает → результат"
[ ] 8. Есть раздел "Что НЕ входит в MVP"
[ ] 9. Ограничения указаны с цифрами (бюджет, сроки, часы в день)
[ ] 10. Критерии успеха измеримы (не "успех", а "20 регистраций за 2 недели")
[ ] 11. НЕТ абстракций ("удобный", "качественный", "современный")
[ ] 12. НЕТ технических решений (не указан фреймворк — это задача SPEC.md)
```

**Если найдена проблема — промпт для исправления:**
```
В PROJECT_IDEA.md раздел [X] слишком абстрактный / размытый / неполный: [что не так].
Перепиши конкретнее: с числами, фактами, примерами.
```

## Workflow: как создать PROJECT_IDEA.md за 20 минут

```
ШАГ 1: Сформулируйте идею в свободной форме
  ↓ (голос/текст — что решаете, для кого, какая боль, какая ценность)
  
ШАГ 2: Откройте Claude AI (web)
  ↓ Загрузите 2 файла:
  • Ваше свободное описание идеи
  • 2-10-PROJECT_IDEA_TEMPLATE.md (этот справочник — как reference)
  
ШАГ 3: Промпт:
  "Структурируй мою идею в PROJECT_IDEA.md по формату из справочника.
   Все 6 блоков заполни конкретикой: с числами, примерами, измеримыми критериями.
   Если чего-то не хватает в моём описании — задай уточняющие вопросы."
  
ШАГ 4: Ответьте на уточняющие вопросы Claude
  ↓ Claude спросит то, что вы не упомянули (цена проблемы, размер аудитории, критерии успеха)
  
ШАГ 5: Получите PROJECT_IDEA.md
  ↓ Claude вернёт готовый документ по 6-блочному формату
  
ШАГ 6: Валидация (этот документ)
  ↓ Пройдитесь по чек-листу выше
  
ШАГ 7: Если нужны правки — попросите Claude
  "В разделе [X] абстракция [Y]. Перепиши конкретнее."
  
ШАГ 8: Передайте PROJECT_IDEA.md в SPEC_WRITER_PROMPT
  ↓ Следующий шаг пайплайна: генерация технической спецификации SPEC.md
```

**Формула пайплайна:**
```
Идея (голос) → Claude → PROJECT_IDEA.md → [SPEC_WRITER_PROMPT] → SPEC.md → [SETUP_GENERATOR] → Claude Code → Готовый продукт
```

**Итог:** вы не написали PROJECT_IDEA.md руками. Вы рассказали Claude идею, валидировали результат, дали правки. Это и есть роль архитектора.
